# 05 — 数据抽象与文件系统

> 对应 `target.md` §3.5 + §4.6

## 讨论范围

xos 不提供传统文件系统。本文讨论 Object Store + Stream 的原生数据模型，三层存储架构的分工，以及如何让用户态文件系统在拥有原生 `mmap` 的同时不牺牲性能。

---

## 为什么不继续用"目录树 + 字节流"

### 字节流文件缺了什么

一个 `fopen()/fread()/fwrite()/fclose()` 的文件：

- 没有结构：应用需要自己解析 JSON/CSV/protobuf/自定义格式
- 没有索引：查找一条记录 = 扫描整个文件（除非应用自己建索引）
- 没有版本：覆盖 = 丢失旧数据（除非应用自己实现 WAL）
- 没有关系：文件之间的联系只存在于应用逻辑中
- 没有事务：写入一半崩溃 = 文件损坏

这些不是"文件系统不该做"的事情——而是现代应用反复重新实现的基础设施。

### 不是替代数据库

xos 的 Object Store 不是 SQLite 的内核版，目标不是替代所有数据库。它的位置在"字节流文件"和"应用数据库"之间：

```
字节流文件               xos Object Store          应用数据库
  │                           │                       │
  无结构                     有结构                   强结构
  无索引                     基础索引                 全索引
  无事务                     原子替换/事务            完整事务
  无版本                     内建版本                 应用级版本
  无关系                     对象引用                 外键/join
```

Object Store = "每个应用都需要的那些数据基础设施，但不需要完整 SQL 引擎"。

## Object Store 设计

### 对象模型

```
Object {
    id: ObjectID,           // 稳定标识（UUID 或内容地址）
    type: ObjectType,       // "document" | "image" | "config" | "blob" | ...
    tags: [Tag],            // 用户标签
    metadata: {             // 系统元数据
        created: Timestamp,
        modified: Timestamp,
        version: Version,
        size: u64,
        mime: Option<String>,
    },
    indices: {              // 自动索引
        "by-tag": ...,
        "by-type": ...,
        "by-date": ...,
    },
    data: Stream<u8>,       // 实际数据，通过 Stream 访问
    references: [ObjectID], // 到其他对象的引用
}
```

### 关键操作

| 操作                         | 语义                    |
| ---------------------------- | ----------------------- |
| `create(type, data)`         | 创建对象，返回 ID       |
| `read(id)`                   | 返回对象元数据 + 数据流 |
| `write(id, data)`            | 原子替换对象数据        |
| `update_meta(id, metadata)`  | 更新元数据              |
| `delete(id)`                 | 删除对象                |
| `query(filter)`              | 按 tag/type/date 等查询 |
| `list_versions(id)`          | 列出所有历史版本        |
| `revert(id, version)`        | 回退到指定版本          |
| `watch(id)`                  | 订阅对象变更事件        |
| `relate(id1, id2, relation)` | 建立对象间关系          |

### 为什么目录树只是兼容投影

对于需要 POSIX 兼容的应用（通过 Compatibility Domain），Object Store 可以投影为目录树：

```
Object Store                              POSIX View (Compat Domain)
────────────────────────────────         ─────────────────────────
object: "user-docs/report-2024"    ←→    /home/alice/documents/report-2024
  type: document                          (普通文件)
  tags: ["work", "2024"]                  (扩展属性 xattr)

object: "user-photos/"              ←→    /home/alice/photos/
  type: collection                        (目录)
  children: [photo1, photo2, ...]         (目录列表)
```

投影层负责维护 POSIX 语义（路径、权限位、目录结构），但原生应用直接用 Object ID 和 tags，不经过路径。

## Stream 抽象

### 与 Object 的分工

- **Object** = 持久状态，可寻址，有版本
- **Stream** = 流动数据，不可寻址（没有"流的位置"），有时序

| 场景          | 使用                     |
| ------------- | ------------------------ |
| 日志输出      | Stream                   |
| 音频/视频播放 | Stream                   |
| 网络连接      | Stream                   |
| 传感器数据    | Stream                   |
| 文件内容读写  | Stream（从 Object 获取） |
| 进程间通信    | Stream                   |

### Stream 的一等语义

xos 的 Stream 不是"加了 `O_NONBLOCK` 的文件描述符"。它原生支持：

- **背压（backpressure）**：生产者不会比消费者快太多
- **取消（cancellation）**：消费者可以取消一个正在等待的读操作
- **批量（batching）**：多条消息合并传输，减少 IPC 往返
- **优先级（priority）**：不同 Stream 有不同的 QoS 等级
- **多播（multicast）**：一个生产者 → 多个消费者

### 批判性审视：Stream 是否在开 Unix 的倒车？

上面的 Stream 场景表覆盖了日志、音视频、网络、传感器、文件内容、IPC——六个看起来完全不相关的领域被统一到了一个抽象下。这自然引出一个尖锐的问题：

> **Unix 把一切压扁成 fd（文件描述符），xos 把一切压扁成 Stream。换了个名字，本质是不是一样？**

这是一个完全合理的质疑。下面诚实面对。

#### Unix fd 的问题到底在哪

Unix "一切皆文件" 的问题不在于 **"IO 操作统一为 read/write"**——这本身是好的工程抽象。问题在于 fd 被强行塞进了不属于 IO 语义的东西：

| Unix 的 fd 用法                      | 问题                                                                                  |
| ------------------------------------ | ------------------------------------------------------------------------------------- |
| `open("/dev/nvme0")` → fd            | 设备控制面和数据面被压成同一个 fd，设备专有操作靠 `ioctl()` 这个万能后门              |
| `socket()` → fd                      | 网络连接被伪装成"文件"，但 `lseek()` 在上面无意义——文件语义和网络语义在 fd 层已经分裂 |
| `open("/etc/nginx/nginx.conf")` → fd | 配置文件的"内容"和"路径身份"被耦合，移动文件 = 改变身份                               |
| `pipe()` → fd                        | 单向字节流，没有消息边界、没有优先级、没有取消                                        |
| `open("/proc/1234/status")` → fd     | 内核状态被暴露为"伪文件"，实际上每次 read 都在动态生成——这是 RPC 假装成文件           |

Unix 用 **一个抽象（fd + 字节流）** 同时扮演了六个角色：持久存储、设备 IO、进程间通信、网络通信、内核接口、配置发现。后四个根本不是"文件"，只是被强行投影到 fd 上。

#### xos 的做法有什么不同

xos **没有**把这些东西压成一个抽象。它有四个不同的一等抽象，各自承担不同的语义：

| 职责                               | Unix 用什么                             | xos 用什么                  | 为什么不同                                                                                         |
| ---------------------------------- | --------------------------------------- | --------------------------- | -------------------------------------------------------------------------------------------------- |
| 持久存储（数据在哪里，是什么版本） | fd (open + path)                        | Object (ID + 元数据 + 版本) | Object 有身份、有索引、有版本，不依赖路径。Object 的内容可以通过 Stream 读写，也可以通过 mmap 访问 |
| 数据流动（字节怎么传输）           | fd (read/write)                         | Stream                      | Stream 有背压、取消、优先级、多播。它不承担"数据是什么"的语义——那是 Object 的事                    |
| 设备控制（谁能操作硬件）           | fd (open /dev/xxx + ioctl)              | 设备能力句柄 + 队列对象     | 控制面和数据面彻底分离。设备能力是权限问题，队列是提交/完成机制——都不是"字节流"                    |
| 服务发现（谁能调用谁）             | 路径约定 (/var/run/xxx.sock) + 环境变量 | Service Graph               | 与数据流动完全无关。服务身份是名字，不是路径                                                       |

**Stream 在 xos 中的定位是纯数据流动抽象**——它不负责"数据是什么"（Object 负责），不负责"谁有权访问"（Capability 负责），不负责"数据从哪个设备来"（设备句柄负责）。

#### Stream 和 fd 的本质区别

| 维度     | Unix fd                                              | xos Stream                                            |
| -------- | ---------------------------------------------------- | ----------------------------------------------------- |
| 身份     | fd 同时是"数据的名字"和"数据的通道"                  | Stream 只是通道，数据的身份由 Object ID 或服务名承担  |
| 生命周期 | fd 由进程持有，dup/fork 隐式传播                     | Stream 是显式创建和传递的，生命周期受 Capability 约束 |
| 取消     | 没有通用取消机制（`close()` 不是取消，是销毁）       | 取消是一等操作，取消后 Stream 仍可继续使用            |
| 背压     | 依赖于 socket buffer 大小和 `SO_RCVBUF` 调整         | 显式背压传导，从消费者到生产者                        |
| 多播     | 需要额外设施（如 `tee()` + pipe，或 `SO_REUSEPORT`） | 内建多播，一个生产者 → 多个消费者                     |
| 批量     | `readv/writev`（仅向量化，不解决往返问题）           | 批量提交，减少 IPC 往返                               |
| 权限     | fd 的权限在 open 时检查一次，之后不再检查            | Stream 的访问受 Capability 句柄约束，可随时撤销       |

#### 诚实地说：仍然有风险

即使有上述区别，Stream 确实面临一个真实的设计风险：**它覆盖了太多场景，可能在实践中变成第二个"万能抽象"**。

具体风险：

1. **语义过载**：日志流的"背压"策略（丢弃旧日志）和音频流的"背压"策略（降低采样率）完全不同。如果 Stream 的 API 试图同时满足两者，要么变成 `ioctl` 式的万能参数堆，要么每个场景都需要不同的 Stream 子类型——这实际上分裂了 Stream 抽象。

2. **TCP 的幽灵**：网络 Stream 和文件内容 Stream 在延迟、丢包、顺序保证上的语义天差地别。把 TCP 连接建模为 Stream 是合理的（它确实是字节流），但把 NVMe 队列也建模为 Stream 就有问题了——NVMe 是提交/完成模型，不是字节流模型。**xos 对此的区分是：NVMe 用队列对象 + 设备句柄，不走 Stream。**

3. **一切皆流 vs 一切皆文件**：Unix 的问题不在"流"这个抽象本身——管道和 socket 确实是流。问题在于它把**不是流的东西**也假装成流（设备配置、文件元数据、内核状态）。xos 必须警惕同样的陷阱：不要因为 Stream 方便，就把设备控制、服务发现、对象元数据查询也塞进 Stream。

#### 设计约束（防止重蹈覆辙）

xos 应该对 Stream 设定以下硬约束：

1. **Stream 只传输不解释**：Stream 不理解它传输的字节的含义。语义（"这是日志"、"这是音频帧"）是上层的事。如果 Stream 开始根据内容类型改变行为，说明抽象已经越界。
2. **Stream 不替代其他抽象**：不能用 Stream 做设备控制（用设备句柄），不能用 Stream 做对象元数据查询（用 Object Store API），不能用 Stream 做服务发现（用 Service Graph）。
3. **Stream 的子类型必须有明确边界**：如果出现 `NetworkStream`、`FileStream`、`SensorStream` 三个子类型且 API 差异很大，那说明它们不应该共用一个抽象。宁可拆开，不要强行统一。

## 三层存储架构

### 内核基座

只做不可替代的硬件仲裁：

- 页框分配与回收（统一页缓存）
- 缺页入口与 Pager 通信
- IOMMU/DMA 映射
- 块设备队列抽象（提交/完成）
- 持久化原语（flush barrier, discard/trim）

不做的事情：

- 不维护 inode/dentry 缓存
- 不实现任何文件系统格式
- 不维护目录结构
- 不实现写回策略

### 用户态存储核心服务

实现"文件系统"的主要逻辑：

- 对象命名、索引、元数据
- 事务协议（WAL/CoW）
- 页缓存策略（预读、写回、冷热分层）
- 崩溃恢复
- 空间分配与 GC
- 快照与克隆

这个服务通过 Pager 接口与内核 VM 协作。参考 [06-pager-and-memory.md](./06-pager-and-memory.md)。

### 直通路径

对于不需要通用存储栈的高性能场景：

```
数据库引擎 ──→ 设备能力句柄 ──→ 块设备队列 ──→ NVMe 设备
                                  (bypass 存储核心服务)
```

权限仍然在内核控制（IOMMU 约束访问范围），但数据路径不经过存储核心服务。

## 性能挑战与对策

### 高频元数据操作

传统内核文件系统的优势：`stat()` 只需要一次系统调用，dentry cache 在全局共享。

xos 的挑战：如果每个 `stat` 等效操作都需要 IPC 到用户态存储服务，延迟会更高。

对策：

1. **元数据推送 + 本地缓存**：存储服务可以推送目录/对象的元数据快照到 Capsule 的本地缓存。大多数 `stat` 等效操作命中本地缓存，不走 IPC。
2. **批量操作**：`list_objects("/photos/")` 一次返回所有子对象元数据，而不是逐个查询。
3. **内核对象 ID 缓存**：对于频繁访问的对象，内核可以在页表中缓存 Object ID → Pager 的映射，减少名字解析的往返。

### 共享页缓存

传统内核的页缓存是所有进程共享的——同一个文件的同一个页只在内核中存一份。

xos 的挑战：如果存储服务在用户态管理页缓存，多个 Capsule 映射同一个文件时如何共享物理页框？

对策：

- 页框池由内核管理（统一内存回收），但页框的"含义"（属于哪个文件的哪个偏移）由存储服务定义
- 内核允许不同 Memory Object 映射到同一个物理页框（当存储服务确认它们确实是同一个内容时）
- 通过 Pager 的 `share_page` 原语合并映射

## 开放问题

1. **Object Store 的查询语言**：简单的 tag/date/type 查询够用吗？还是需要一个轻量查询语言（如 GraphQL 子集）？
2. **大对象的流式部分更新**：修改一个 10GB 视频文件的一个字节，是否需要重写整个对象？是否需要类似"对象 + 增量补丁"的模型？
3. **Object 的引用完整性**：Object A 引用 Object B，但 B 被删除了。如何处理？类似外键约束还是弱引用？
4. **跨设备同步冲突**：如果两台设备离线修改了同一个 Object，冲突解决策略是什么？（与 §4.8 账户同步相关）
