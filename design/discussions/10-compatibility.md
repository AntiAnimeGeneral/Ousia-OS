# 10 — Linux 兼容与去中心化账户

> 对应 `target.md` §4.7 + §4.8

## 讨论范围

兼容旧生态是新 OS 无法回避的问题。本文讨论 Linux 兼容域的设计、与原生系统的交互边界，以及去中心化账户体系的定位和设计约束。

---

## Linux 兼容：既要生态，又不污染原生层

### 为什么兼容层必须是隔离的，不是嵌入的

历史上有很多"兼容 Unix 的新 OS"走了嵌入路线——在原生 API 中直接提供 POSIX 兼容接口（如 `open()`, `read()`, `fork()` 等）。这导致：

- 原生设计被 POSIX 语义拖累（如所有 API 必须接受路径字符串）
- 应用无法区分"它在原生系统上"还是"它在兼容层上"
- 开发者习惯性地使用 POSIX API 而非原生 API
- 最终原生抽象被架空，系统退化为"又一个 Unix-like"

xos 的解决方案：**兼容域是虚拟机，不是 API 适配层**。

### 兼容域网关

网关负责在原生空间和兼容域之间转换：

| 资源类型 | Linux 兼容域视角                    | 原生空间视角                 |
| -------- | ----------------------------------- | ---------------------------- |
| 文件     | `/home/alice/docs/` (路径)          | Object "user-docs" (对象 ID) |
| 窗口     | X11/Wayland socket                  | 原生窗口系统协议             |
| 网络     | socket fd                           | 网络能力句柄                 |
| 剪贴板   | X11 selection / Wayland data-device | 原生剪贴板服务               |
| 设备     | `/dev/dri/renderD128`               | 设备能力句柄                 |

网关不尝试"完美映射"——它提供足够好的转换，让大多数 Linux 应用在兼容域中可运行。

### 隔离保证

兼容域：

- 运行在独立的 VM/container 中
- 受资源配额约束（默认比原生应用更受限）
- 不能绕过能力模型访问原生资源
- 崩溃不影响原生系统

## 去中心化账户：设计约束，不做过度设计

### 为什么需要账户

如果 xos 未来需要：

- 跨设备数据同步
- 软件商店和付费分发
- 发布者身份验证
- 端到端加密通信

那么"谁是谁"必须有一个系统级的答案。它不需要中心化服务器，但需要一个可移植的身份模型。

### 最小设计

```
身份 (Identity) = 设备无关的标识符
    │
    ├── 公私钥对（ed25519 或类似）
    ├── 可验证声明 (Verifiable Claims)
    │   ├── "我是 alice@example.com"（由 example.com 签名证明）
    │   ├── "我是这个设备的拥有者"（由设备 TPM/安全元件证明）
    │   └── "我可以发布软件包"（由发布者网络签名证明）
    │
    └── 设备绑定
        ├── 主设备（持有主私钥）
        └── 次级设备（持有派生密钥，可被主设备撤销）
```

### 对内核的影响

最小化：内核只需要支持一种新的能力类型——**身份句柄**（Identity Capability）。

```
IdentityHandle {
    identity_id: IdentityID,
    claims: Set<Claim>,
    proof: Signature,    // 由身份私钥签名
}
```

一个 Capsule 可以持有身份句柄，在与其他服务交互时出示。内核不需要理解"账户"概念——那在用户态处理。

### 对 Package Cell 的影响

发布者身份绑定到 Cell 的签名：

```
Cell {
    publisher: IdentityID,
    signature: Signature,   // 由发布者私钥对内容哈希签名
}
```

系统安装 Cell 时：

1. 检查签名是否有效
2. 检查发布者是否被信任（用户配置的信任策略）
3. 如通过 → 安装
4. 如不通过 → 拒绝（或提示用户手动确认）

## 不做的事情

xos 第一阶段**不**实现：

- 完整的去中心化账户系统（用户注册、跨设备绑定、密钥恢复）
- 区块链/DID 文档存储
- 去中心化软件商店
- 付费分发基础设施

只预留：

- 身份句柄类型（内核能力模型）
- 发布者签名字段（Package Cell 格式）
- 信任策略配置接口

## 开放问题

1. **兼容域的性能开销**：虚拟化级别的 Linux 兼容会有多少性能损失？对于图形密集型应用（如游戏），是否可行？
2. **Wayland/X11 的映射复杂度**：Linux GUI 应用的窗口协议如何映射到 xos 原生窗口系统？是否需要类似 XWayland 的中间层？
3. **身份恢复**：如果用户丢失主设备（和主私钥），如何恢复身份？社交恢复？备份密钥？
4. **信任根**：谁来签名"发布者可以发布软件包"这个声明？需要类似 CA 的层次结构还是 web of trust？
