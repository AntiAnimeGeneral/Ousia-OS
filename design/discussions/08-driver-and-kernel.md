# 08 — 内核原语与驱动框架

> 对应 `target.md` §4.3 + §4.9

## 内核边界：最小原语层

内核只保留不可替代的全局仲裁：调度+执行等级、地址空间+页表+缺页入口、中断+时钟、IPC+能力句柄、IOMMU+DMA、MMIO 授权、Pager、启动句柄注入、设备复位与接管。

内核绝对不做：文件系统格式实现、驱动代码、名字解析、包管理、策略决策、服务发现。

**辩证：为什么不把驱动放回内核？** 业界四条路线证明了纯用户态驱动的可行性，各自解决了不同层面的质疑：

| 路线                         | 证明了什么                                                                                                                 |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **Windows WDDM**             | GPU 驱动可拆为"用户态厂商主驱动 + 内核调度/内存管理层"。厂商热路径、着色器编译器全部在用户态；内核只做显存仲裁和上下文切换 |
| **Apple DriverKit**          | 驱动可作为受签名、受授权、可升级的用户态扩展交付。崩溃不影响内核，系统仍掌握隔离和恢复                                     |
| **Fuchsia Driver Framework** | 驱动主机、驱动索引、组件化分发和进程级隔离可成为系统默认模型。不是特例，是架构                                             |
| **VFIO/SPDK**                | 在有 IOMMU 和 DMA 所有权模型的前提下，用户态可获得接近裸机的设备访问性能。性能不是放回内核的理由                           |

xos 不是"选其中一条"——是综合四条的设计：WDDM 的用户态/内核分工 + DriverKit 的签名和隔离 + Fuchsia 的组件化生命周期 + VFIO 的 IOMMU 基座。

## 用户态驱动架构：四个运行实体

- **Driver Manager**：设备发现、绑定、生命周期管理、权限授予、崩溃恢复编排
- **Driver Index**：根据设备属性（VID/PID/Class Code）匹配驱动包。本质是一个键值查询服务
- **Driver Host**：承载用户态驱动实例的受限 Capsule。有设备控制权，但仍然受沙盒约束（不能随意访问文件系统或网络）。驱动崩溃不影响内核——内核收回设备、隔离现场、通知 Driver Manager 编排恢复
- **Kernel Hardware Mediation Layer**：最小可信硬件控制面——IOMMU 页表、MMIO 授权、设备复位（FLR/总线复位/电源复位）、中断路由、电源仲裁。不包含任何设备特化逻辑

## 闭源驱动的工程路径

**xos 不强迫厂商开源。** 现实是 GPU/NPU/高速网卡厂商不会开源着色器编译器后端、性能优化和硬件 workaround。xos 接受这个现实，设计了一条工程上可接受的纯用户态接入路径：

- 驱动作为签名的 Driver Package 分发，不进入内核树
- 厂商核心逻辑（着色器编译器、命令缓冲生成、专有优化）在用户态共享库中，由 Driver Host 加载
- 只要遵守稳定驱动 ABI、能力模型和资源治理约束，厂商可保留全部专有实现
- 系统仍可审计其能力使用、限制其权限、崩溃收敛、版本管理和回滚
- Driver Host 提供隔离边界——即使厂商 C 代码有漏洞，攻击面被限制在 Driver Host 的 Capsule 沙盒内

**这恰恰是纯用户态驱动架构的卖点，不是妥协。** 它比"把闭源二进制塞进内核"安全得多——而 Linux 的 NVIDIA 驱动正是这么做的。

## 高性能 IO 原语

- **设备能力句柄**取代 `/dev` 节点。应用不需要知道设备"在哪里"——能力由系统根据 Cell 声明自动授予
- **队列对象**：通用的提交/完成抽象，支持 doorbell 和批量通知
- **mmap 设备 BAR/doorbell/共享环**：用户态直接写快路径，热路径不经过系统调用
- **中断 event 对象**和**低抖动轮询对象**并存：高负载时用轮询（避免中断开销），低负载时切回中断（省功耗）
- **驱动主机共置**：强耦合驱动在同一进程内通信，可选 in-process transport 避免内核往返
