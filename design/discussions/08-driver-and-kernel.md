# 08 — 内核原语与驱动框架

> 对应 `target.md` §4.3 + §4.9

## 讨论范围

xos 内核是"最小原语层"，不是"驱动容器"。本文讨论内核的职责边界、用户态驱动架构的四大运行实体、硬件仲裁层的设计，以及闭源驱动的工程路径。

---

## 内核职责的"最小"边界

### 内核必须做的事

只有这些事情必须在内核中发生：

| 职责                 | 原因                             |
| -------------------- | -------------------------------- |
| 调度与执行等级       | 全局抢占优先级只有内核能做       |
| 地址空间管理（页表） | 硬件 MMU 的操作必须在特权级      |
| 缺页入口             | 缺页异常只能在内核处理           |
| 中断路由             | 硬件中断只能内核接收和转发       |
| 时钟/定时器          | 需要硬件定时器支持               |
| IPC 句柄传递         | 能力句柄不可伪造，必须由内核管理 |
| IOMMU 页表           | 硬件 IOMMU 的操作必须在特权级    |
| DMA 映射与所有权     | 防止设备 DMA 到任意物理内存      |
| MMIO 授权            | 防止用户态直接访问任意设备 MMIO  |
| 能力句柄管理         | 句柄的创建/派生/撤销必须在内核   |
| 设备复位与接管       | 崩溃驱动后收回设备需要硬件仲裁   |
| Pager 故障检测       | 超时检测和映射终结必须在内核     |

### 内核绝对不做的事

| 禁止事项         | 原因                           |
| ---------------- | ------------------------------ |
| 文件系统格式实现 | 所有格式解析在用户态           |
| 驱动代码         | 所有设备特化逻辑在用户态       |
| 名字解析         | 这是名字服务的职责             |
| 包管理           | 这是 Package Cell 管理器的职责 |
| 策略决策         | 内核只提供机制，策略在用户态   |
| 服务发现         | 这是 Service Graph 的职责      |
| 健康检查         | 顶多提供心跳超时，不定义"健康" |

## 用户态驱动架构的四大实体

### 1. Driver Manager

```
职责:
- 设备发现（PCI 枚举、设备树解析、ACPI 表遍历）
- 设备绑定（根据 VID/PID/Class Code 找驱动）
- 生命周期管理（加载、启动、停止、卸载驱动）
- 权限授予（将设备能力句柄授予 Driver Host）
- 恢复编排（驱动崩溃后决定重启还是放弃）
```

### 2. Driver Index

```
职责:
- 维护"设备 → 驱动包"的映射数据库
- 根据设备属性（VID, PID, Revision, Class Code, Subsystem ID）
  匹配最优驱动包
- 支持优先级和签名验证
```

Driver Index 本质上是一个键值查询服务，查询键是设备属性，值是驱动包的内容地址。

### 3. Driver Host

```
职责:
- 加载和运行一个或多个驱动实例
- 提供进程级隔离（一个驱动崩溃不影响其他 Driver Host）
- 提供驱动 SDK 的运行时环境
- 为同 Host 内的多个驱动提供 in-process transport
```

Driver Host 是一个受限制的 Capsule——它的能力比普通应用多（有设备控制权），但仍然受沙盒约束（不能访问文件系统，不能随意连网）。

### 4. Kernel Hardware Mediation Layer

```
职责:
- IOMMU 页表管理（授予/撤销 DMA 映射）
- MMIO 区域授权（授予/撤销 MMIO 访问）
- 设备复位（FLR / 总线复位 / 电源复位）
- 中断路由（设备中断 → 用户态中断事件对象）
- 电源仲裁（设备的 D0-D3 状态转换）
- 错误隔离（设备 AER/UE 错误不扩散）
```

这是最小的硬件可信基——它不包含任何设备特化逻辑。

## 设备能力句柄

### 设备打开流程

```
1. Capsule 请求访问 GPU
   → Package Cell 声明中标记了 "gpu:render" 能力需求

2. 系统检查：
   - 该 Capsule 被授予了 GPU 能力吗？
   - 该 GPU 有可用的驱动吗？
   - 该 GPU 的队列/引擎还有容量吗？

3. 如果通过：
   - Driver Manager 找到 GPU 的 Driver Host
   - 内核创建 DeviceCapability{device: gpu0, rights: RENDER|COMPUTE}
   - 将该能力句柄授予 Capsule

4. Capsule 使用该句柄：
   - 映射 GPU BAR 的 doorbell 页 → 直接写命令
   - 从命令队列读取完成通知
   - 整个过程不再需要系统调用（热路径）
```

### 关键设计：设备句柄不是路径

在 Unix 中：`open("/dev/dri/renderD128")` → 返回 fd

在 xos 中：设备能力句柄由系统根据 Cell 声明自动授予。应用不需要知道设备"在哪里"。

## 闭源驱动的工程路径

### 为何闭源驱动是现实

GPU 厂商（NVIDIA, AMD, Intel, Apple, Qualcomm）不会开源他们的着色器编译器后端、性能优化和硬件 workaround。这是行业现实。xos 的设计必须接受这一点，同时不因闭源而妥协安全性。

### xos 的方案

```
┌─────────────────────────────────────────┐
│ Driver Host (用户态)                      │
│  ┌─────────────────────────────────┐    │
│  │ 厂商闭源 .so (C/C++/PTX)        │    │
│  │ - 着色器编译器                   │    │
│  │ - 命令缓冲区生成                 │    │
│  │ - 性能优化逻辑                   │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │ xos Driver SDK (Rust safe API)   │    │
│  │ - 设备能力句柄封装               │    │
│  │ - 队列提交/完成管理              │    │
│  │ - 中断注册                       │    │
│  │ - 错误处理                       │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
          │ 稳定 ABI
          │ (能力句柄传递, 共享内存, 队列对象)
┌─────────────────────────────────────────┐
│ Kernel Hardware Mediation Layer         │
│ - IOMMU 授权                            │
│ - MMIO BAR 映射                         │
│ - 中断路由                              │
│ - 复位控制                              │
└─────────────────────────────────────────┘
```

闭源厂商只需要：

1. 将驱动编译为共享库
2. 链接 xos Driver SDK
3. 打包为签名的 Driver Package
4. 提交到 Driver Index

不需要开源任何东西。

### 安全边界

- 厂商代码在用户态运行，崩溃不影响内核
- 厂商代码只能访问被明确授予的设备（IOMMU 约束）
- 厂商代码的网络/文件系统访问受 Capsule 沙盒限制
- 系统的驱动录制回放工具可以捕获厂商驱动的行为用于审计

## 高性能 IO 原语

### 队列对象

xos 不要求厂商驱动使用特定的队列格式（如 NVMe SQ/CQ 或 virtqueue）。它提供通用的队列原语：

```
QueueObject {
    type: SubmissionQueue | CompletionQueue,
    memory: MappedMemory,       // 共享内存中的队列区域
    doorbell: Option<MMIORegion>, // 门铃寄存器（如果有）
    event: EventObject,         // 用于通知（替代中断轮询）
    batch_size: usize,          // 批量提交大小
}
```

### 中断 vs 轮询

xos 不强制驱动使用中断或轮询。它提供两种模型：

- **中断对象**：驱动注册中断事件，低负载时休眠等中断
- **轮询对象**：驱动请求一个专用 CPU 核做低抖动轮询（对于 NVMe/网卡这类高吞吐设备）

驱动可以在两种模式之间动态切换——高负载时用轮询，低负载时切回中断。

## 开放问题

1. **Driver Host 崩溃后，设备的硬件状态如何保证干净？** 内核的复位协议能处理所有设备类型吗（PCIe FLR 不是所有设备都支持）？
2. **多个 Driver Host 共享一个物理设备**（如多个应用同时使用 GPU）：由谁负责 GPU 上下文切换？内核还是用户态 GPU 调度器？
3. **稳定 ABI 的版本管理**：驱动 ABI 的向后兼容性是业界难题。xos 能承诺多长时间的不变 ABI？如果必须破坏，迁移路径是什么？
4. **驱动签名的信任根**：谁有权签名驱动包？系统自带的可信 CA 列表？用户是否可添加自己的信任根？
