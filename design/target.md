项目自造术语和重新定义过的设计术语见 [glossary.md](./glossary.md)。除非特别说明，文档中的 Portal、Operation、Continuation、Communication Fabric 等词都是 Ousia OS 的设计术语，不指代某个现有系统的专有技术。

# 设计目标

让一切回归正确的抽象，以现代软件栈的真实痛点为起点，重新设计一个安全、可演进、面向异构硬件的系统。

这个系统的核心立场不是"复刻 Unix 再修补"，而是从头定义更合理的默认模型：

- Unix 继承下来的最低公分母抽象不是系统真相。
- 软件交付默认可复现、可验证、可回滚。
- 应用、服务和依赖的生命周期默认由操作系统统一管理，而不是交给脚本、约定和用户手工维护。
- 调度默认保护前台交互与关键任务活性，不因后台作业失去响应。
- 性能是一级约束，系统边界不能以牺牲关键路径性能为代价。
- 软件运行环境默认随交付单元封装，不靠环境变量和 shell 初始化脚本拼装。
- 进程默认无权限，只有显式授予的能力。
- 系统接口默认异步，不把阻塞调用当成常态。
- 数据默认有类型和语义，而不只是路径下的一串 blob。
- 兼容性通过隔离层提供，而不是反向污染整套原生抽象。

## 0. 反 Unix 抽象立场

Ousia OS 说"反 Unix"，不是反工具化、反文本接口，也不是反工程简洁性。我们反对的是：把历史妥协形成的最低公分母抽象继续当成现代系统的最终真相。

Ousia OS 明确反对以下默认前提：

- 反对"一切皆文件"：服务、对象、流、设备、能力、软件包不是同一种东西，不应被强行压扁成路径下的字节流。
- 反对"路径字符串同时承担命名、定位、发现和授权"：这些职责应该拆开，由服务名、对象标识和能力句柄分别承担。
- 反对"进程只是继承环境变量和全局命名空间的一段程序"：运行单元应该是带声明、带能力、带资源预算的 Capsule。
- 反对"用户身份天然等于权限边界"：权限应绑定对象和操作，通过 Capability 显式授予、转交和回收。
- 反对"同步阻塞 + fork/exec/pipe 是系统组合的最终模型"：原生组合模型应是异步消息、流、对象引用和服务图。
- 反对"设备能力只能通过内核态特化驱动和 `/dev` 暴露"：设备控制面与厂商驱动逻辑应解耦，后者默认停留在用户态。
- 反对"包管理、服务管理、依赖治理只是系统外的附加工具"：它们本来就是操作系统的核心职责。

正确的抽象不该追求"把一切伪装成同一种东西"，而应让不同事物以正确的系统语义存在，再在边界上提供兼容投影。

## 1. 现代软件栈的核心痛点

### 1.1 依赖、安装与分发失控

今天的软件栈最大的问题不是"功能不够多"，而是软件交付链条过于脆弱：

- 依赖树深且不可控，一个上游变更就可能破坏整套环境。
- 系统包管理、语言包管理、容器镜像彼此割裂，最终由用户承担复杂度。
- 动态链接、全局库、环境变量、路径约定共同组成隐式状态，行为不可预测。
- 很多软件要求用户手工修改 PATH、LD_LIBRARY_PATH、PKG_CONFIG_PATH、bashrc、profile 等环境，安装成功依赖大量外部前置条件。
- 卸载经常不干净，会在 shell 配置、缓存、符号链接和全局目录中留下脏状态。
- 升级、回滚、并存、多版本隔离缺少统一模型。
- 经典钻石依赖问题（A 依赖 C v1 和 C v2）在现有系统中要么靠手动解决，要么靠容器全量封装回避，缺乏系统级的内建仲裁。

### 1.2 Unix 的默认权限模型过于宽松

传统进程模型默认拥有大量环境可见性：

- 默认可见整个文件系统命名空间。
- 默认可发起网络连接。
- 默认可读取大量系统状态。
- 权限通常是"用户身份继承"，而不是"操作能力授予"。

这会导致沙盒成为补丁，而不是第一性设计。

### 1.3 同步阻塞式接口与粗糙调度不适合现代系统

现代硬件和现代软件都是高度并发的，但大量系统接口和调度模型仍然继承过时前提：

- 一个计算密集型或 IO 密集型作业就可能打满 CPU、GPU、内存带宽或 IO 队列，让 GUI、输入、音频和桌面服务一起失去响应。
- 实时任务、交互任务、前台服务、批处理任务、后台维护任务缺少清晰的一等语义和隔离模型。
- 难以表达取消、超时、背压、批处理。
- 异步通常只是用户态框架包装，不是内核和服务的统一语义。
- 面向 IO、UI、网络、设备、分布式任务时，同步接口会把复杂性转嫁到上层。

系统必须把"任何单个作业都不该轻易拖死整机交互"当作基本正确性要求，而不是性能优化。

### 1.4 文件系统抽象落后于现代数据使用方式

"目录树 + 字节流文件"是通用最低公分母，但不是现代最佳抽象：

- 程序真正关心的是对象、关系、版本、索引、权限和同步语义。
- 路径字符串既承担命名又承担定位，难以稳定引用与安全授权。
- 元数据贫弱，应用只能重复造数据库、缓存和索引系统。
- **配置文件是一个错误的抽象**：配置不必是文件。`.conf` / `.yaml` / `.toml` / `.env` 格式碎片化，解析错误只在运行时暴露，无法校验、无法回滚、无法原子更新多个相关配置项。现代 web 服务早已不靠文件管理配置——它们用数据库和配置中心。OS 应该原生提供类型安全、可校验、可回滚的配置管理，而不是把配置强塞进字节流文件再让每个应用重复实现解析器。

### 1.5 兼容性与原生设计经常互相拖累

如果把历史兼容直接压进原生 API，原生设计会被历史包袱锁死。但如果完全不兼容，生态又无法起步。

### 1.6 异构硬件已经不是例外

CPU 大小核、GPU、NPU、DSP、SmartNIC 都是现代系统的常态，但现有操作系统仍主要围绕"均质 CPU + 外设"组织调度与资源模型。电源管理同样缺乏跨异构资源的统一抽象，大小核调度、设备休眠和功耗预算分配往往依赖厂商私有的 out-of-tree 方案。

### 1.7 抽象边界不能以关键路径性能为代价

"所有东西都搬去用户态"并不自动等于更先进。如果边界划分不考虑缓存局部性、上下文切换、IPC 往返、锁竞争和故障语义，系统会在关键路径上付出过高成本：

- 高频 IO 和元数据路径可能被多次拷贝、上下文切换和跨地址空间协调拖慢。
- 某些状态一旦损坏或中断，并不适合依赖"杀掉再拉起"恢复。
- 过度教条地坚持用户态分层，可能让文件系统、内存管理协作、设备快路径等核心能力在实践中变得不可用。

因此，内核态和用户态的边界必须由可信性、可恢复性和真实性能成本共同决定，而不是由意识形态决定。

## 2. 系统设计总纲

### 2.1 总目标

Ousia OS 要做的不是"更快的 Linux"，而是建立一套新的默认秩序：

- 软件以声明式单元交付。
- 应用、服务与依赖由系统统一编排和治理。
- 运行环境随软件单元一起声明和封装。
- 权限以能力对象授予。
- 调度以交互可用性和关键任务保活为前提，再追求吞吐。
- 文件系统语义与驱动主逻辑必须停留在用户态，内核只提供通用原语。
- 系统以服务图而非全局命名空间组织。
- 数据以带语义对象存储和流处理组合表达。
- 兼容层以虚拟化和代理方式承接旧生态。

### 2.2 顶层设计原则

1. 默认安全优先于默认方便
2. 显式声明优先于隐式约定
3. 机制与策略分离
4. 不把 Unix 最低公分母当作系统真相，原生抽象优先
5. 异步优先，同步只是包装层
6. 交互可用性与关键路径保活优先于单纯吞吐
7. 运行环境属于软件单元，不属于用户 shell
8. 驱动逻辑与高层策略必须在用户态；FS 采用混合态——元数据修改在用户态，元数据查询热路径在内核缓存（FS 推送维护）。内核不承载可扩展性会被锁死的特化实现
9. 内核提供通用原语、隔离、仲裁和热路径元数据缓存，不直接承载厂商驱动
10. 可复现、可回滚、可审计是系统级能力
11. 性能通过更强的原语和更短的热路径保证，而不是通过把逻辑塞回内核
12. 内核与可信基础服务必须使用内存安全语言实现，关键安全属性的规约应可机器检查
13. 故障模型遵循"能恢复则恢复，不能恢复或代价太大则 let it crash"：系统设计为 crash-tolerant，状态损坏时优先快速失败并由上层监督者重启，而非在不可靠状态下继续运行

### 2.3 实现语言策略

Ousia OS 从零开始，没有历史 C 代码负担。语言选择直接影响安全性和开发效率：

- **内核**：使用 Rust 或不带 GC 的内存安全语言。内核中的 `unsafe` 块必须显式标注、可审计，并有形式化理由。参考 Asterinas 的框架化安全实践，将 unsafe 封装在经形式化验证的核心原语中，上层内核逻辑在 safe 子集中编写。
- **用户态基础服务**：优先使用 Rust 或其他提供强类型和内存安全的语言。对于策略注入和受限扩展场景，可采用 WASM 作为受限执行载体。
- **驱动**：驱动 SDK 应提供安全的 Rust 绑定；对于必须使用 C 的闭源厂商驱动，由 Driver Host 提供隔离边界，C 代码不进入内核地址空间。
- **形式化目标**：第一阶段的合理目标不是全系统形式化验证，而是对能力传递、缺页处理、IOMMU 映射等安全关键路径做机器可检查的规约（参考 seL4 的经验，但不需要一开始就做到同等深度）。

## 3. 核心系统抽象

为了避免设计发散，Ousia OS 需要围绕少数几个一等公民抽象展开。

### 3.1 软件单元：Package Cell

软件的基本交付单位不是"散落在系统各处的一组文件"，而是一个可验证的软件单元，暂称 Package Cell。它至少包含：

- 内容地址或签名标识
- 依赖声明（精确到版本约束的声明式依赖图）
- 运行环境声明
- 能力需求声明
- 数据模式声明
- 服务暴露声明
- 生命周期钩子
- 兼容层需求

系统负责安装、激活、并存、升级、回滚和垃圾回收。用户和开发者不直接操作共享的全局库目录，也不应依赖手工修改 shell 配置来完成软件启用。

**依赖冲突策略**：允许多版本并存（每个 Package Cell 在自己的命名空间中看到自己声明的依赖版本），避免钻石依赖问题成为系统级阻塞点。第一阶段通过限制约束语言、允许不兼容版本并存、使用确定性选择规则，避免把系统安装变成全局唯一版本的通用约束求解问题。解析结果锁定在内容地址图谱中，确保可复现。跨 Capsule 通信时，接口兼容性由 Service Graph 的版本协商机制处理，而非通过全局符号版本强制统一。

### 3.2 运行单元：Sandbox Capsule

所有进程都在沙盒中运行，沙盒是默认模型而不是可选特性。一个 Capsule 的边界包括：

- 可见服务集合
- 可访问数据对象集合
- 可使用设备能力集合
- 可发起的网络能力集合
- 可继承或可转交的能力句柄

一个应用是否能访问资源，不由"它是谁"决定，而由"它当前拿到了哪些能力"决定。

### 3.3 权限单元：Capability

系统权限不再基于全局 uid/gid 语义扩散，而是基于可组合的能力对象：

- 能力可被创建、授予、拆分、转交、回收。
- 能力应尽量绑定对象与操作，避免粗粒度授权。
- 能力传播路径可审计。
- 用户交互授权最终转化为能力令牌或句柄。

**能力撤销的实现边界**：完整的能力撤销（包括级联拆分撤销和跨 Capsule 转发追踪）是 capability 系统中最难的问题之一。Ousia OS 采用分阶段策略：

- **第一阶段**：仅支持直接授予和内核可见派生链上的撤销。内核维护每个能力句柄的创建链，撤销时沿链失效。暂不支持完整级联和任意跨 Capsule 转发追踪。
- **第二阶段**：引入能力代理（Capability Broker）服务，对拆分和转发做注册追踪，支持非实时、best-effort 的级联撤销通知。硬件资源能力（如 DMA 映射）的撤销走 IOMMU 页表同步的原子路径，不依赖代理。
- 第一阶段的设计预留能力链追踪所需的元数据字段，避免后期改 ABI。

**待决疑点：能力转发与撤销语义**。如果一个可转发能力在第一阶段绕过内核可见链路被多次转交，原始授予者无法可靠撤销所有间接持有者。为避免安全语义被误读，第一阶段应进一步区分“不可转发/租约型能力”和“可转发能力”：敏感资源默认使用不可转发或短租约能力；需要跨 Capsule 转发的能力必须通过内核可见转移或 Capability Broker 登记后，才承诺更强撤销语义。这个边界在正式 ABI 前必须定稿。

### 3.4 系统组织单元：Service Graph

系统不是围绕一个全局文件树组织，而是围绕一张服务图组织：

- 名称解析到服务或对象，而不是默认解析到路径。
- 系统服务通过受控接口交互。
- 服务发现、版本协商、权限校验是统一机制。
- 进程之间优先通过消息、对象引用和流进行通信。

**Bootstrap 设计**：服务图的初始启动不依赖名字解析——那会导致鸡与蛋问题。内核在创建第一个用户态进程时，通过启动句柄（Bootstrapping Handle）直接注入一组初始能力：包括一个初始名字服务的能力句柄、一个初始内存对象的能力句柄、以及一个到内核自身服务（如创建 Capsule、分配地址空间）的通道。名字服务自身是第一个被启动的用户态服务，其名字就是它的能力句柄本身——不需要"解析"自己。后续所有服务发现都从这组初始能力派生。

### 3.5 数据单元：Object Store + Stream

文件系统不再只暴露"路径 -> 字节流"。原生数据层至少包含两类基础抽象：

- Object Store：面向持久对象、元数据、版本、索引、关系
- Stream：面向 IO、日志、媒体、网络、传感器和实时数据流

目录树可以作为兼容视图存在，但不应成为内核和原生应用的唯一真相。

### 3.6 计算单元：Compute Domain

CPU、GPU、NPU 等异构资源需要统一纳入调度和权限模型：

- 任务可声明算力类型、执行等级、时延目标、吞吐目标、功耗目标。
- 系统按 Compute Domain 和执行等级路由到适合的执行后端。
- 关键交互路径和实时任务必须与后台作业隔离，避免前台失活。
- 驱动和运行时通过统一资源描述暴露能力。

### 3.7 内存对象：Pager-backed Memory Object

如果 Ousia OS 希望把文件系统主逻辑尽量放在用户态，同时又原生支持 `mmap`、按页缺页、共享映射和系统级虚拟内存治理，那么内核必须把"可分页内存对象"做成一等公民：

- 内核负责页表、页框、缺页入口、映射、回收和最终访问控制。
- 用户态服务可为内存对象供应页面、失效页面、查询脏页、写保护页面、参与回写。
- 文件映射不应退化为每次读写都走消息 RPC，而应通过 Pager-backed Memory Object 与 VM 子系统直接协作。
- 共享映射、写时复制、回写和内存压力下的回收需要统一落在同一套 VM 原语上。

这类对象是"用户态文件系统 + 原生虚拟内存"能否成立的关键。

**Pager 崩溃与恢复模型**：Ousia OS 对用户态 Pager 服务崩溃采取"能恢复则恢复，不能恢复或代价太大就 let it crash"的立场。具体契约如下：

- 内核为每个 Pager-backed Memory Object 维护一个 Pager 句柄和一个故障超时。当缺页请求发送后在超时内未收到响应，内核判定 Pager 故障。
- Pager 故障时，内核立即终结该 Pager 句柄关联的所有 Memory Object 映射。持有这些映射的 Capsule 收到一个不可恢复的页面故障信号（`MEMORY_OBJECT_LOST`），进程被终止或进入预定义的崩溃处理路径，不会无限阻塞在缺页上。
- 不尝试内核态自动修复或透明备援切换——这些逻辑的成本和复杂度不应在内核中。恢复（如果可能）由 Pager 监督服务在用户态执行：重新实例化 Pager、重新建立映射、通知受影响的 Capsule 重新获取数据。
- 崩溃一致性由用户态存储核心服务的事务日志保证。内核不负责回放日志或猜测用户态的意图。Pager 崩溃时，未提交到事务日志或未完成持久化确认的脏页可以丢失；已经通过 `fsync`/`msync` 或等价持久化屏障确认的修改必须由存储服务保证可恢复。普通 buffered write 和 mmap dirty page 的语义需要明确区分“已接收”“已入日志”“已持久化”三个阶段，不能把每次写入都默认解释为同步落盘。

## 4. 目标设计

### 4.1 操作系统统一管理应用、服务与依赖

Ousia OS 的立场是：应用、服务、依赖和运行环境不应在系统之外野蛮生长，而应纳入操作系统的原生管理边界。

目标不是做另一个包管理器，而是让"软件交付 + 服务编排 + 依赖治理 + 运行环境生成"一起成为操作系统能力。

应满足：

- 应用、服务和其依赖以统一的软件单元建模
- 安装不污染全局环境
- 安装、运行和构建环境由声明生成，不依赖 PATH、bashrc、profile 等全局配置
- 服务的启停、升级、权限、资源预算和健康状态由系统统一管理
- 不同版本可并存
- 升级可原子切换
- 卸载必须干净且可验证
- 回滚可一键完成
- 依赖解析结果可复现（确定性算法 + 内容地址锁定）
- 软件来源可验证
- 权限请求在安装期和运行期都可见
- 钻石依赖通过多版本并存 + Service Graph 版本协商解决，不成为系统级阻塞点

这要求 Package Cell、运行环境描述、能力声明、内容寻址存储和系统服务编排统一设计。

### 4.2 所有进程默认沙盒化

这不是安全附加层，而是系统基本运行方式。

每个 Capsule 默认：

- 看不到宿主全局命名空间
- 拿不到任意网络访问权
- 不能直接访问设备
- 不能读取未授权数据对象
- 不能通过环境变量和共享路径隐式逃逸边界

需要兼顾开发体验，因此必须提供声明式调试能力、临时授权、开发模式配置和完整审计日志。

### 4.3 最小原语内核 + 能力权限系统

Ousia OS 在这条边界上的立场应当明确而强硬：文件系统主逻辑和所有驱动都必须运行在用户态。内核不是驱动容器，也不是文件系统实现层；内核只保留不可替代的全局仲裁、硬件隔离和虚拟内存原语。

内核必须保留并做强的基础原语包括：

- 调度与执行等级
- 地址空间管理、页表和缺页入口
- 中断、时钟与 EventPort/WaitSet 等等待对象
- Communication Fabric 原语（Portal、Operation、Continuation、SharedQueue、Fence/Timeline）与句柄传递
- 能力对象与句柄管理
- IOMMU、DMA 映射与设备隔离
- 用户态可用的 MMIO/队列/内存注册/doorbell 原语
- Pager-backed Memory Object 支持
- 元数据快取（路径→OID 和 OID→元数据的热路径缓存，由用户态 FS 推送维护，只读）
- 启动句柄注入原语（用于 Service Graph bootstrap）
- early path / panic path 的最小硬件通路

这些原语共同构成一个最小可信硬件控制面（Hardware Core）。它不是传统驱动层：设备特化逻辑、厂商策略和复杂协议默认留在用户态；内核只保留隔离、复位、早期启动、故障收敛和可验证 fast-path assist 所需的通用机制。

在这些原语之上，Ousia OS 应提供一套面向纯用户态驱动的一等抽象，让高性能设备路径不再依赖内核驱动：

- 设备能力句柄：明确授予某个服务对设备、队列、BAR、DMA 区域的控制权
- 队列对象：用于提交/完成、doorbell、共享环和批量通知
- 内存注册对象：用于固定页、DMA 可达内存、共享页池和零拷贝缓冲区
- 中断/轮询对象：允许服务在低抖动轮询和受控中断间切换
- 复位与接管协议：当用户态驱动崩溃时，内核可收回设备、隔离现场并交给接替服务恢复

这些对象的目标是围绕资源授权、队列所有权和 fast-path assist 建立稳定 substrate，而不是一开始就冻结一个庞大的 per-device syscall ABI。

这套设计应明确吸收当前最佳实践：

- Windows WDDM 路线证明了图形和计算驱动可以采用"用户态厂商主驱动 + 内核调度/内存管理层"的分工
- Apple DriverKit 路线证明了驱动可以作为受签名、受授权、可升级的用户态扩展交付
- Fuchsia Driver Framework 路线证明了驱动主机、驱动索引、组件化分发和进程级隔离可以成为系统默认模型
- VFIO/SPDK 路线证明了在 IOMMU、DMA 映射和设备所有权模型足够强时，用户态可获得接近裸机的设备访问性能

因此，Ousia OS 对高性能用户态驱动的立场应是：

- 厂商驱动必须作为用户态组件交付，不进入内核地址空间
- 内核只承载最小可信硬件仲裁层：设备发现、隔离、复位、电源、调度仲裁、内存映射和 DMA 所有权控制
- 厂商驱动的热路径、编译器后端、着色器栈、队列调度策略、专有优化和用户 API 适配层全部留在用户态
- 内核与用户态驱动之间使用稳定、版本化、可审计的设备 ABI，而不是大量不稳定的内部符号

这样做的目标不是弱化内核，而是把内核从"塞满策略的实现层"变成"为用户态高性能系统服务提供基座的原语层"，同时彻底把扩展性锁定在用户态。

### 4.4 调度模型必须保证交互性与实时性

现代系统不能把所有工作负载都当作"公平竞争的普通进程"。Ousia OS 需要把执行等级作为显式系统语义：

- 支持实时、交互、前台服务、批处理、后台维护等不同执行等级
- GUI、输入、合成器、音频、窗口系统和核心系统代理必须有保底预算与更强抢占保护
- 当系统资源紧张时，优先降级后台吞吐，而不是让前台失去响应
- 长任务必须支持取消、暂停、配额、背压和资源上限
- CPU 之外，GPU、内存带宽、IO 队列、NPU 等也需要纳入同一资源治理模型

目标不是禁止高负载，而是保证任何单个作业都不能轻易拖死整机交互。

**电源管理作为调度的一等维度**：执行等级天然携带功耗预算语义。交互和实时任务不仅有延迟保证，还隐含"允许提升频率/电压"的预算；后台批处理则携带"优先能效核、限制功耗上限"的约束。系统级的电源状态（设备休眠、CPU idle 状态、GPU 降频）必须与调度器共用同一份资源模型，而不是独立的外挂策略。Compute Domain 的功耗目标声明同时影响调度决策和设备电源状态选择。

### 4.5 所有 API 第一性支持异步

系统调用、服务调用、设备访问都要有统一的异步语义：

- Future/Promise 只是语言表象，内核内部需要有原生等待对象和取消模型
- 所有长耗时操作都可取消、可超时、可组合
- 支持背压和批处理
- 消息、流、事件、计时器使用统一等待机制

同步 API 可以作为语法糖存在，但不能成为底层真实模型。

**统一通信基座：Communication Fabric**。Ousia OS 不把通信模型压成单一“同步 IPC”或单一“异步 Channel”。内核应提供一组正交通信原语族：Portal（服务入口能力）、Operation（请求生命周期对象）、Continuation（受限回复权）、EventPort/WaitSet（完成与事件聚合）、SharedQueue（受治理的共享队列）、MemoryObject/IOBuffer（大数据和 DMA 可达内存）、Fence/Timeline（跨队列同步）。

这套原语应支持三条最低成本路径：

- **小控制消息**：走 Portal fast call，尽量直接交接，不进入通用内核消息队列，不分配大缓冲。
- **异步请求**：走 Operation + Continuation + EventPort，由 OS 统一管理 cancel、timeout、late reply、pending quota、priority/deadline propagation 和 completion routing。
- **高频数据面**：走 SharedQueue / IOQueue + MemoryObject/IOBuffer + Event/Fence/Doorbell，用 bounded ring、批量提交和内核旁路避免逐消息 syscall 或 IPC。

协议格式、IDL、method id、业务版本协商、Future/Promise 语言绑定和服务框架路由留在用户态；但通信生命周期、等待、取消、背压、调度预算传播、观测和撤销必须是统一系统语义。详见 [09-communication-fabric.md](./core/09-communication-fabric.md)。

**异步与 mmap 的张力**：Ousia OS 的"异步优先"立场和 `mmap` 的同步缺页语义之间存在天然张力——访问一个未映射页会导致线程 stall，直到用户态 Pager 响应。这不是设计矛盾，而是两种不同访问模型的共存：

- 显式 IO（read/write/stream）默认走异步路径，适合大部分应用逻辑。
- `mmap` 是为特定场景（零拷贝文件访问、共享内存数据库、高性能图形管线）提供的受控同步捷径。使用 `mmap` 的 Capsule 应当声明其执行等级，调度器在缺页 stall 期间保护其优先级预算，同时对其 Pager 的响应时间做监控——一个持续慢响应的 Pager 会被视为故障并按 §3.7 的模型终止。
- 应用可通过预取（prefetch）原语主动将页面提前填充，避免在热路径上触发缺页 stall。

### 4.6 全新的文件系统设计

Ousia OS 的原生数据层目标不是替代所有数据库，而是提供比传统文件系统更高一级的系统抽象：

- 对象拥有稳定标识，不依赖路径字符串做唯一引用
- 元数据、标签、索引、版本是内建能力
- 权限授予绑定对象和操作类型
- 支持事务性更新或至少原子对象替换
- 支持本地优先和多设备同步
- 目录树仅作为兼容投影层之一

可把它理解为"对象存储、内容寻址、索引系统、流式 IO"的组合，而不是单一文件树。

在实现路径上，Ousia OS 不教条式地坚持"FS 全部在用户态"或"FS 全部在内核"。正确的边界是**混合态**，从一开始就确定：

- **内核空间**：页缓存（物理页框管理）、元数据快取（路径→OID 映射和 OID→{size,type,mtime} 的热路径缓存，由用户态 FS 推送维护，只读）、块 I/O 队列、Pager 接口（缺页捕获+页表填充）、文件句柄管理
- **用户态 FS 服务**：元数据修改（create/unlink/rename）、事务/日志（WAL, CoW）、空间分配、压缩/加密、缓存策略（预读窗口、回写时机、冷热分级）、GC、快照

**为什么元数据修改在用户态**：`create`/`unlink`/`rename` 最终要命中稳定存储，I/O 延迟（µs~ms）主导总延迟，IPC 往返开销（ns~µs）可忽略。

**为什么元数据查询在内核有缓存**：`stat`/`open` 在热路径上每秒数千次。内核元数据缓存（FS 推送维护）让缓存命中时不经过 IPC、不切换到用户态，目标是接近传统 dentry/inode cache 的热路径成本。缓存未命中时回退到 IPC 查询 FS。

这不是"先用户态不行再加内核缓存"的渐进方案。内核元数据缓存是**核心设计**，不是后期优化。没有"先试用户态再迁回内核"的试错空间——边界必须一开始就做对。

这意味着 Ousia OS 的文件系统目标不应是"把 `mmap` 降级成普通 RPC 读写"，而是让用户态文件系统原生参与虚拟内存：

- 文件可被映射为 Pager-backed Memory Object
- 缺页由内核捕获，但页面内容可由用户态存储核心服务供应或确认
- 脏页跟踪、写保护、失效和回写必须有显式原语，不能靠隐式约定
- 页缓存应是内核管理的统一页框池，但其预读、写回、冷热分层和对象级策略可由用户态服务主导

如果这些原语设计得足够好，那么事务、页缓存协作、崩溃一致性和核心元数据路径不只是"可以"在用户态实现，而是应当在用户态实现，同时仍然享有原生 `mmap`、零拷贝、共享映射和系统级虚拟内存治理。

SPDK、io_uring、硬件无锁队列、零拷贝共享内存这类技术，确实可能在特定负载上显著反超传统内核路径，但前提非常具体：

- 设备模型适合用户态直驱或低开销提交完成队列
- 工作负载以高并发、深队列、顺序流或批处理为主
- 应用可以接受轮询、内存固定、专用核占用或更复杂的资源治理方式
- 数据路径可以从减少拷贝、减少系统调用、减少锁竞争中稳定受益

它们并不意味着"用户态文件系统会自动更快"。对于下面这些场景，用户态架构必须拿出比传统内核更强的原语设计与运行时组织，否则很容易失去优势：

- 低延迟且低队列深度的通用桌面 IO
- 频繁的小块元数据操作与跨进程缓存共享
- 需要统一页缓存、内存回收、写回、权限校验和系统级 QoS 的场景
- 一旦状态损坏就会影响系统恢复和全局一致性的核心存储路径

因此，Ousia OS 的答案不应是"关键时刻把文件系统塞回内核"，而是"让内核为纯用户态文件系统提供足够强的原语"。只要用户态驱动抽象、Pager 抽象、内存注册和设备隔离做对，文件系统主逻辑就必须停留在用户态，而不牺牲性能。

目标不是只押注某一种"最先进技术"，而是同时保住三件事：

- 通用负载下的稳定低尾延迟
- 专用负载下的极限吞吐与低开销
- 整个系统的一致性、可恢复性与可治理性

### 4.7 Linux 兼容

兼容的目标是获取生态，不是牺牲原生抽象。

建议将 Linux 兼容定义为单独的 Compatibility Domain：

- 通过轻量虚拟化或类 Hyper-V 分区承载 Linux 用户态与内核 ABI
- 与原生系统通过代理网关交换文件、窗口、网络、剪贴板、设备能力
- 兼容域默认仍受能力和资源配额约束

原则是：兼容层向旧生态让步，原生层不向旧抽象让步。

### 4.8 去中心化账户

如果系统未来需要跨设备同步、付费分发、身份认证和可信发布，账户体系必须提前设计。

目标不是强依赖某个中心化服务器，而是定义一个可移植身份模型：

- 身份与设备分离
- 身份可多端绑定
- 发布者身份可验证
- 用户数据同步授权可审计
- 允许中心化托管实现，但协议不绑定单一服务商

它可以参考联邦身份、DID、硬件密钥和可验证声明，但不应把链上化当成默认前提。

**定位说明**：账户体系属于第 2 层（平台服务），不是内核能力。它对内核的影响仅限于：身份句柄作为一种特殊能力类型可跨 Capsule 传递，用于授权决策。第一阶段不实现完整的去中心化账户，但内核的能力模型预留身份句柄的支持，避免后期改 ABI。

### 4.9 利于驱动开发

驱动开发必须被当作一等工程体验设计：

- 提供稳定的驱动 SDK 和设备模型
- 驱动必须在用户态运行
- 驱动权限最小化
- 驱动崩溃可隔离、可恢复
- 提供录制回放、仿真设备、协议抓取和一致性测试工具

驱动框架本身也应该围绕异步事件和能力授权设计。

Ousia OS 应明确采用"纯用户态驱动 + 最小内核硬件仲裁层"的架构，这既更利于高性能，也更符合现实厂商生态，尤其是 GPU、NPU、高速网卡和专用加速器这类常见闭源驱动场景。

这套架构至少需要以下运行实体：

- Driver Manager：负责设备发现、绑定、生命周期管理、权限授予与恢复编排
- Driver Index：负责根据设备属性、厂商 ID、类代码和能力需求匹配驱动包
- Driver Host：负责承载一个或多个用户态驱动实例，提供进程级隔离和崩溃边界
- Kernel Hardware Mediation Layer：负责最小可信硬件控制面，包括 IOMMU、MMIO 授权、DMA 所有权、复位、电源和中断路由；它不是驱动，只是不带设备特化策略的通用仲裁层

对闭源厂商，Ousia OS 的目标不是强迫其开源，而是给出一条工程上可接受的纯用户态接入路径：

- 驱动必须作为签名的 Driver Package 分发，而不是打补丁进内核树
- 厂商核心逻辑可位于用户态共享库或服务中，由 Driver Host 加载
- 只要遵守稳定驱动 ABI、能力模型和资源治理约束，厂商可以保留专有实现
- 系统仍可对其进行审计、限权、崩溃收敛、版本管理和回滚

为了避免用户态驱动性能被框架本身吃掉，Ousia OS 需要提供以下最佳实践级能力：

- 设备 fd/句柄而不是全局节点文件，避免隐式共享和权限扩散
- 基于 IOMMU 的 DMA 所有权模型，明确谁可映射哪些内存
- 每队列或每引擎独占的提交/完成对象，避免热路径上的全局锁
- 支持 `mmap` 的设备 BAR/doorbell/共享环映射，让用户态可直接写快路径
- 中断 event 对象和低抖动轮询并存，让不同设备选择合适模式
- 驱动主机共置能力：可让强耦合驱动在同一进程内通信，避免不必要的内核往返
- 可选的 in-process transport，用于同宿主内驱动与文件系统/图形栈的低开销通信

这套架构尤其适合 GPU 类驱动。当前行业最佳实践本身就趋向于：

- 用户态放置图形 API 适配层、编译器、命令缓冲生成、着色器优化和大量厂商私有逻辑
- 内核态仅保留显存所有权、调度仲裁、上下文切换、超时恢复、显示链路和最小安全控制所需的通用仲裁原语，而不承载厂商驱动逻辑

对 Ousia OS 来说，这说明一个重要方向：不应该把"支持闭源厂商"理解为对设计妥协，恰恰相反，一个良好的纯用户态驱动架构本来就应该要求厂商把全部驱动逻辑放在用户态，同时让系统继续掌握隔离、恢复和资源治理。

### 4.10 良好设计的系统组件框架

系统服务、后台代理、策略模块、可插拔逻辑需要统一组件模型：

- 明确生命周期
- 明确资源声明
- 明确能力边界
- 明确热更新或热插拔语义

这里可以引入类似 WebAssembly 或 eBPF 的受限执行环境，但用途要明确：

- 用于策略注入
- 用于过滤、观测、转换
- 用于受限扩展

不应把它当作通用逃逸层，否则会重新引入不可控复杂度。

### 4.11 面向全领域但只支持现代硬件

Ousia OS 不应该在第一阶段就试图覆盖所有场景，而应该定义统一内核与多配置产品线：

- 个人设备版：强调应用模型、UI、同步、兼容性
- 服务器版：强调隔离、编排、可观测性、资源治理
- 实时版：强调确定性调度、低抖动、受限 API 子集
- 车机/嵌入版：强调裁剪、可靠更新、功能安全边界

共用同一套核心抽象，但允许不同配置裁剪和认证路径。

**硬件最低要求**：Ousia OS 第一阶段只支持 64 位 ARM（AArch64）和 x86-64，要求 UEFI 启动、IOMMU（或 ARM SMMU）、标准化中断控制器（GICv3+ 或 xAPIC/X2APIC）。明确不支持：BIOS/Legacy 启动、32 位 CPU、无 IOMMU/SMMU 的系统、传统 PCI 不带 ACS 的设备。这条硬边界确保能力模型、设备隔离、IOMMU-based DMA 所有权和用户态驱动框架不需要为老旧硬件做妥协设计。

### 4.12 异构硬件是第一公民

现代系统必须把异构资源纳入统一调度和开发模型：

- 任务可表达是否可卸载到 GPU/NPU
- 数据移动成本必须进入调度决策
- 设备内存与主存的可见性模型需要统一描述
- 权限和隔离不能因为是加速器就失效
- 功耗预算作为 Compute Domain 的一等属性跨 CPU/GPU/NPU 统一分配

这要求驱动框架、运行时、调度器和编程模型共同设计，而不是后期补丁。

### 4.13 内核自身更新模型

微内核虽然小，但仍需要安全更新。Ousia OS 的内核更新采用 A/B 启动分区模型：

- 内核镜像与系统关键服务（第 1 层服务）打包为一个可原子切换的 System Image。
- 更新时写入非活跃分区，验证签名和哈希，然后通过固件或 bootloader 原子切换到新版本。
- 如果新版本启动后健康检查超时未通过，bootloader 自动回退到旧分区。
- 内核更新的安全性与 Package Cell 的更新共享同一套签名验证链和回滚策略，不引入独立的更新机制。

### 4.14 构建系统与测试基础设施

一个全新 OS 的工程化基础设施本身就是一等设计要素：

- **构建系统**：基于 LLVM 工具链。内核和用户态服务使用同一个构建系统（如 Bazel 或类似的内容寻址构建系统），确保依赖关系可复现、增量构建可靠。第一阶段的交叉编译目标为 aarch64-unknown-Ousia OS 和 x86_64-unknown-Ousia OS。
- **测试策略**：
  - 内核单元测试：在宿主系统上直接运行的内核逻辑测试（不依赖硬件）
  - 用户态服务集成测试：在模拟的 Capsule 环境中运行
  - 驱动模拟测试：通过回放录制的设备交互进行回归测试（§4.9 的录制回放工具）
  - 形式化模型检查：对能力传递、缺页处理和 IOMMU 映射等关键安全路径进行模型检查，不需要一开始就做到全系统形式化验证

## 5. 非目标

为了防止设计失焦，下面这些不应成为第一阶段目标：

- 不追求兼容传统 POSIX 作为原生接口
- 不追求支持老旧硬件和历史 BIOS 生态（见 §4.11 硬件最低要求）
- 不追求一开始就自研所有语言运行时和全部桌面应用
- 不追求在原生抽象未稳定前开放大量不受控内核扩展接口
- 不追求完整的能力级联撤销和跨 Capsule 转发追踪（预留设计，第一阶段仅支持直接撤销）
- 不追求去中心化账户体系的完整实现（预留身份句柄类型，平台服务阶段再做）

## 6. 建议的系统分层

### 6.1 第 0 层：微内核

提供调度、执行等级、内存管理、Communication Fabric、能力句柄、时钟、中断、IOMMU 仲裁、启动句柄注入和 Pager 故障检测等核心原语。

内核自身不包含名字解析、不包含文件系统逻辑、不包含驱动代码。

### 6.2 第 1 层：基础系统服务

Bootstrap 顺序：内核启动后注入初始能力句柄 → 启动名字服务（第一个用户态服务，其身份就是其句柄本身） → 启动 Capsule 管理器 → 以此为基础依次启动其余服务。

包括：

- 名字服务（Service Graph 的根，由内核启动句柄直接注入）
- 进程与 Capsule 管理器
- 资源治理与 QoS 服务
- 对象存储服务
- 网络服务
- 设备管理服务
- 驱动管理器
- 驱动索引服务
- 驱动宿主
- 日志与观测服务
- 身份与授权服务
- Pager 监督服务（监控用户态 Pager 健康、编排崩溃后恢复）

### 6.3 第 2 层：平台服务

包括：

- Package Cell 管理器
- 图形与窗口系统
- 数据同步服务
- 策略引擎
- 兼容域网关
- 账户与身份同步服务（后期）

### 6.4 第 3 层：应用与兼容环境

包括：

- 原生应用
- 系统组件
- Linux 兼容域
- 开发者工具链

## 7. 第一阶段落地建议

如果要把设计从理念推进到工程，第一阶段不要试图"一步到位"。更合理的顺序是：

1. **微内核最小原语**：任务、地址空间、Communication Fabric、能力句柄（含直接撤销）、基础抢占调度、启动句柄注入
2. **用户态名字服务和 Capsule 管理器**：验证 Service Graph bootstrap 和能力授予流程
3. **Pager-backed Memory Object + 最小 Pager 原型**：验证缺页通路和 Pager 崩溃/"let it crash" 语义
4. **最小 Capsule 模型和能力授予流程**：验证沙盒默认模型和能力显式传递
5. **极小的对象存储服务**：先验证"路径不是唯一真相"，基于用户态 Pager 实现
6. **Package Cell 原型**：跑通安装、启动、回滚、多版本并存和钻石依赖解决
7. **驱动框架最小原型**：验证设备能力句柄、IOMMU 授权和用户态 MMIO 映射通路
8. **最后再接入兼容层和更复杂的设备/图形栈**

## 8. 设计判断标准

未来每一个重要设计都可以用下面几个问题过滤：

1. 这个设计是否消除了隐式全局状态？
2. 这个设计是否让权限变得更显式、可审计、可回收？
3. 这个设计是否默认支持异步、取消和背压？
4. 这个设计是否保证前台交互与关键任务不会被普通作业饿死？
5. 这个设计是否消除了对全局环境变量、shell 配置和手工安装步骤的依赖？
6. 这个设计是否强化了 Package Cell 和 Capsule 这两个核心抽象？
7. 这个设计是否把兼容性限制在边界上，而不是污染原生接口？
8. 这个设计是否对异构硬件（含电源管理）友好？
9. 这个设计是否遵循"能恢复则恢复，不能恢复就 let it crash"的故障模型？
10. 这个设计的故障模式是否可测试、可观测、可诊断？

如果答案多数是否定的，那它大概率仍然在重复旧系统的问题。

## 9. 参考项目

Ousia OS 的设计思路受以下项目的启发，但不复刻其中任何一个：

- **Fuchsia**：微内核 + 组件化 + 能力模型 + 用户态驱动框架
- **Asterinas**：Rust 内核框架，将 unsafe 封装在经形式化验证的核心原语中，上层内核逻辑在 safe 子集中编写
- **seL4**：形式化验证的微内核，能力系统的正确性证明
- **Windows WDDM**：用户态厂商主驱动 + 内核调度/内存管理层的分工
- **Apple DriverKit**：受签名、受授权、可升级的用户态驱动扩展
- **SPDK / io_uring**：高性能用户态 IO 路径的工程验证
