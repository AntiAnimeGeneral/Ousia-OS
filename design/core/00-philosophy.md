# 00 — 反 Unix 立场与设计总纲

> 展开 [target.md](../target.md) 中的设计立场与目标约束。

## 讨论范围

本文展开讨论 Ousia OS 的核心哲学立场——为什么"反 Unix"是必要的起点，以及在此基础上建立的设计总纲和原则。

---

## 0.1 "反 Unix"到底反什么？

### 不是反什么

首先要澄清误解。Ousia OS 不反对：

- **工具化思想**：Unix 的"do one thing well"是好的工程哲学，Ousia OS 继承这个精神但用不同方式表达
- **文本接口**：结构化文本、管道组合是有效的交互模式
- **工程简洁性**：简洁是美德，但简洁不等于"把所有东西压成同一种抽象"
- **C 语言或 shell 本身**：它们是好工具，问题在于它们变成了系统边界的唯一定义方式

### 是反什么

Ousia OS 反对的是七条具体的默认前提。下面逐条展开讨论。

#### 1. 反对"一切皆文件"

**问题本质**：文件接口（open/read/write/close + 字节流）是最低公分母——任何东西都能假装成它，但几乎所有东西都不该只是它。

| 对象     | 被压扁成文件后丢失了什么               |
| -------- | -------------------------------------- |
| 服务     | 版本协商、健康检查、生命周期、能力发现 |
| GPU 设备 | 命令队列、doorbell、DMA 映射、故障恢复 |
| 网络连接 | 背压、取消、流控、QoS 等级             |
| 能力句柄 | 可传递性、可拆分性、可撤销性           |
| 软件包   | 依赖图、版本约束、运行环境声明         |

**Ousia OS 的替代方案**：让不同事物以正确的系统语义存在。服务有服务发现协议，设备有能力句柄，网络有流对象，能力有句柄语义。

**tree view 与兼容翻译**：Ousia 保留一等 tree view，用于人类导航、作用域、挂载和兼容生态。旧软件通过 Compatibility Domain 看到 POSIX 风格路径翻译；原生应用可以同时使用路径、ObjectHandle、OID、标签和能力句柄。

#### 2. 反对"路径字符串同时承担命名、定位、发现和授权"

这是一条被严重低估的设计缺陷。在 Unix 中，`/usr/bin/python` 同时是：

- **命名**：这个东西叫 python
- **定位**：它在 `/usr/bin/` 这个位置
- **发现**：通过 `$PATH` 搜索找到它
- **授权**：通过文件权限位（rwx）控制谁能执行它

这四个职责耦合在一起导致：

- 移动文件 = 改名（符号链接只是补丁）
- 两个同名文件不能在同一目录（命名冲突）
- 授权粒度只有 owner/group/other 三级
- 发现依赖 `$PATH` 这个隐式全局状态

**Ousia OS 的拆分方案**：

- **命名**：Service Graph 中的服务名，与位置无关
- **定位**：能力句柄，内核维护的对象引用
- **发现**：名字服务，通过服务名解析到能力句柄
- **授权**：能力模型，绑定对象 + 操作类型

#### 3. 反对"进程只是继承环境变量和全局命名空间的一段程序"

传统 Unix 进程继承太多隐式状态：

- 环境变量（`PATH`, `HOME`, `LD_LIBRARY_PATH`...）
- 文件系统命名空间（整个目录树可见）
- 用户身份（uid/gid）
- 当前工作目录
- 文件描述符（从父进程继承）

这些隐式继承导致：

- 进程的行为不可从自身声明预测
- 沙盒化需要额外工具（chroot, namespaces, containers）
- "在我的机器上能跑"成为常态

**Ousia OS 的 Capsule 模型**：运行单元带声明（依赖、能力需求、资源预算），不带隐式继承。

#### 4. 反对"用户身份天然等于权限边界"

uid/gid 模型的根本问题是：**权限绑定人，而不是绑定操作**。

- 一个进程以用户 alice 身份运行 → 它能做 alice 能做的一切
- 无法表达"这个进程只能读这一个文件，不能读 alice 的其他文件"
- 无法表达"临时授予，用完收回"

**Ousia OS 的 Capability 模型**：权限是对象，不是身份属性。可以显式授予、拆分、回收。

#### 5. 反对"同步阻塞 + fork/exec/pipe 是系统组合的最终模型"

fork/exec 的问题：

- fork 的写时复制语义在现代多线程程序中是灾难（fork-safety）
- exec 替换整个地址空间，启动成本高
- pipe 是单向字节流，没有消息边界、没有优先级、没有取消
- 同步阻塞让单线程程序卡死整个交互

**Ousia OS 的替代**：异步消息、流对象、对象引用、服务图。fork/exec/pipe 在 Compatibility Domain 中提供，不作为原生模型。

#### 6. 反对"设备能力只能通过内核态特化驱动和 /dev 暴露"

`/dev` 把设备压扁成文件节点，丢失了设备控制面和数据面的所有结构。内核驱动模型把厂商逻辑锁死在内核地址空间，一次崩溃 = 全系统 panic。

**Ousia OS 方案**：纯用户态驱动 + 内核硬件仲裁层。

#### 7. 反对"包管理、服务管理、依赖治理只是系统外的附加工具"

apt/dpkg、pip、npm、docker 彼此不知道对方的存在。用户承担了跨层依赖协调的全部复杂度。

**Ousia OS 方案**：Package Cell 是操作系统的原生管理单元，系统负责整个生命周期。

## 2.2 顶层原则的深入讨论

### 原则 13：let it crash

这是 Ousia OS 最激进的设计选择之一。传统 OS 内核在几乎所有错误路径上都尝试恢复——但恢复代码本身就是最常见的 bug 来源。

Ousia OS 的立场：

- **能恢复则恢复**：可预测的、代价可控的错误（如缺页 → Pager 响应）走正常恢复路径
- **不能恢复就 let it crash**：Pager 故障、驱动崩溃、状态损坏 → 快速失败，由上层监督者重启
- **代价太大就 let it crash**：即使理论上可恢复，但恢复逻辑的复杂度和风险 > 重启成本时，选择崩溃

这个立场影响整个系统的错误处理设计：

- 内核代码中的错误路径大幅减少 → 更少 bug
- 用户态服务的崩溃是"正常事件"而非"紧急事件"
- 监督者（supervisor）是一等设计，不是运维脚本
- 系统必须设计为 crash-tolerant：任何单个组件崩溃不应级联

### 原则 12：内存安全语言

为什么不是 C：

- Linux 内核中 ~70% 的 CVE 是 memory safety bug
- 从零开始的新内核没有理由继续使用 C
- Rust 的 ownership 模型天然适合"能力句柄不能隐式复制"的语义

但需要明确：

- 不是所有 unsafe 代码都要形式化验证——那是 seL4 级别的目标
- 第一阶段的目标是：unsafe 块可审计、有形式化理由文档
- Asterinas 的框架化思路（unsafe 封装在核心原语中）是合适的参考

## 开放问题

1. **"反 Unix"这个措辞本身会不会成为社区接纳的障碍？** 是否应该用更建设性的表述，如"post-Unix"或"重新定义系统抽象"？
2. **let it crash 的边界在哪里？** 内核自身崩溃意味着全系统重启，这需要重启时间在亚秒级。这个目标可行吗？
3. **原则 12（内存安全语言）和原则 13（let it crash）之间有没有张力？** ——如果系统设计为 crash-tolerant，memory safety 的重要性是否相对降低？
